1.反序列化漏洞

1.1 漏洞成因

序列化说通俗点就是把一个对象变成可以传输的字符串,比如下面是一个对象:

反序列化unserialize()

就是把被序列化的字符串还原为对象,然后在接下来的代码中继续使用。

pikachu中反序列化题目漏洞成因：

主要原因在于使用了 unserialize 函数。unserialize 函数用于将经过序列化的字符串转换回 PHP 的数据结构。这可能导致安全风险，因为恶意用户可以构造精心设计的序列化字符串来触发不受预期的行为，如代码注入、对象注入等。

在这里，unserialize 可能会执行非预期的代码，因为它可以将恶意构造的数据恢复为 PHP 对象，并在执行时利用对象中的内容。这可能导致任意代码执行，包括但不限于服务器端攻击。

1.4 漏洞防御

  1. 避免使用 unserialize： 尽量避免在程序中使用 unserialize 函数，特别是处理来自用户的输入。如果需要处理序列化数据，考虑使用 JSON 或 XML 等更受控制的数据交换格式。
  2. 数据验证和过滤： 对于用户输入的数据，进行严格的验证和过滤。验证输入数据的格式、长度、类型等，并确保只有符合预期的数据才能通过。比如，限制输入只能为特定格式的字符串或数字。
  3. 白名单验证： 如果需要处理用户提供的类名或对象，应该使用白名单验证，限制可接受的类名或对象类型。只有预先定义的类可以被反序列化。

2.XXE漏洞

2.1 漏洞成因

问题出现在 simplexml_load_string() 函数上。当它尝试解析通过 $_POST['xml'] 参数提供的 XML 时，并没有正确地阻止外部实体引用的包含。这可能允许攻击者包含外部实体，可能导致敏感信息泄露甚至远程代码执行。

2.4 漏洞防御

  1. 过滤输入内容：验证并清理 XML 输入，确保其中不包含意外或恶意内容。
  2. 禁用外部实体加载：使用 LIBXML_NONET 替代 LIBXML_NOENT。这可以彻底阻止加载外部实体。
  3. 采用白名单方法：只允许解析特定的元素或属性，拒绝其他内容。

分析漏洞成因：

  1. SQL 注入风险：
    • 虽然在使用 $username=escape($link, $_GET['username']); 对 $_GET['username'] 进行了转义处理，但这里仍然存在 SQL 注入的风险。最好使用参数化查询或预处理语句来防止 SQL 注入攻击。
  2. 缺少权限验证：
    • 代码中仅仅验证了是否有传入的 username，但并没有进行登录状态或身份验证的检查。这可能导致用户能够通过传入任意 username 获取其他用户的信息。
  3. 错误处理不足：
    • 缺少对于数据库查询失败或者未找到用户信息的处理。如果没有找到用户信息，或者数据库查询出现问题，应该有相关的错误处理和信息提示。
